# Semantic Search

## Intent

As an IR case worker or manager, I want to search across all cases and their content using natural language — so I can quickly find relevant past queries, similar cases, and precedents without needing exact keywords.

## Scope

Search index covers:
- `Case.subject`
- `Case.ai_summary`
- `Email.body_html` (all emails linked to cases)
- `CaseNote.content` (internal notes)

## Appian Semantic Search

Uses Appian's OOTB Semantic Search capability. No external search infrastructure required.

### Index configuration

- Entity: `IQH_Record_Case` (primary) with related `Email` and `CaseNote` content included.
- Embeddings generated by Appian at index time and updated on record write.
- Re-indexing triggered on: case creation, new email attached, note added.

### Search UI

- Search bar available in the site header (global) and within the case list view.
- Results returned as ranked case cards showing: reference number, subject, category, status, matched snippet.
- Results respect row-level security — case workers only see cases they have access to.

## Appian objects

- `IQH_SemanticSearch_Config` — Appian semantic search configuration object
- `IQH_Interface_SearchResults`
- `IQH_Interface_SearchBar`

## Acceptance criteria

- [ ] Search works with natural language queries (e.g. "investor asking about redemption fees")
- [ ] Results are ranked by semantic relevance
- [ ] Results respect row-level security
- [ ] Search returns results from case subject, AI summary, email body, and notes
- [ ] Matched text snippet is shown in each result card
- [ ] Search is accessible from the global site header
- [ ] Index is updated within a reasonable time (< 5 minutes) after a new case or email is created

## Open questions

- Should search support filtering by category/status alongside the semantic query?
- Should there be a "similar cases" feature on the case workspace that auto-searches for related past cases?
- What is the expected index size at steady-state (number of cases per year)?
